# 🧠 AI Interview Practice Platform – Backend Architecture

## 📌 Overview

This platform provides AI-driven technical interview practice with:

* AI-generated questions
* AI-based rubric evaluation
* Real-time proctoring enforcement
* Risk scoring
* Historical performance tracking
* Storage-efficient lifecycle management

The system is built using:

* Node.js (JavaScript)
* Express
* PostgreSQL (Neon)
* Raw SQL (no ORM)
* Google AI Studio API
* JWT authentication
* Structured AI prompt validation

---

# 🏗 System Architecture

The backend is organized into domain modules:

modules/
├── user/ → Authentication & identity
├── interview/ → Question generation & lifecycle
├── session/ → Session state management
├── proctoring/ → Live security enforcement
└── analytics/ → History & performance tracking


We follow:
routes → controller → service → queries (raw SQL)

Controllers are thin.
Business logic exists only in services.
All DB interaction is via parameterized SQL.

---

# 🔄 Session Lifecycle

Each interview session follows this state machine:

CREATED
↓
IN_PROGRESS
↓
COMPLETED
↓
EVALUATED
↓
(CLEANED internally)


Or if malpractice occurs:

IN_PROGRESS → TERMINATED

---


State transitions are strictly enforced in backend.

# 🧠 AI Workflow

## 1️⃣ Question Generation (1 API Call)

When session starts:

* Backend calls AI with structured prompt.
* AI returns exactly 10 questions.
* Questions are stored in `ai_questions`.
* Session status → `IN_PROGRESS`.

Prompt Structure:

Rules:

* Strict JSON
* Exactly 10 questions
* No explanation

---

## 2️⃣ Answer Submission & Evaluation (1 API Call)

After user completes answers:

* Answers stored temporarily in `answers`.
* Backend sends full Q&A JSON to AI.
* AI returns:

  * Per-question rubric scoring
  * Overall score
  * Strengths
  * Weaknesses
  * Improvement plan

Prompt:


## 3️⃣ Rubric Aggregation

Before deleting answers:

We compute:

* technical_avg
* depth_avg
* clarity_avg
* problem_solving_avg
* communication_avg

These values are stored in `session_analysis`.

---

# 🔐 Live Proctoring & Security Engine

Frontend detects violations and sends structured events.

Backend:

* Accumulates violations in `session_security`
* Computes weighted score
* Applies thresholds
* Terminates session if limits exceeded

---

## Violation Scoring Logic

Each event has backend-defined weights.

Example:

* NO_FACE → high weight
* MULTIPLE_FACES → high weight
* TAB_VISIBILITY → critical weight
* HEAD_YAW → low weight

Live score is updated atomically.

If thresholds exceeded:

* Session status → TERMINATED
* termination_reason stored
* Evaluation not performed

# 🧩 Database Structure

## 👤 users

Stores authentication and role information.

---

## 🔐 refresh_tokens

Manages refresh token lifecycle.

---

## 🎯 interview_sessions

Core session metadata.

Stores:

* user_id
* skill
* experience_level
* status
* overall_score
* risk_score
* risk_level
* termination info
* timestamps

---

## ❓ ai_questions (Temporary)

Stores AI-generated questions.

Deleted after evaluation or termination.

---

## ✍️ answers (Temporary)

Stores user answers and per-question scoring.

Deleted after evaluation or termination.

---

## 📊 session_analysis (Permanent)

Stores final evaluation summary.

Fields include:

* overall_score
* rubric averages
* strengths
* weaknesses
* improvement_plan
* malpractice_flag
* malpractice_summary (JSON snapshot)

This table preserves user history and dashboard data.

---

## 🛡 session_security (Temporary Live Engine)

Used only during active session.

Stores:

* total_score
* violation_count
* per-event aggregated counters
* heartbeat timestamp
* termination flag

Deleted after session ends.

---

# 🧮 Risk Score System

After session completion:

* Risk score derived from session_security totals.
* Risk level classified as:

  * LOW
  * MEDIUM
  * HIGH

Stored in interview_sessions.

# 🔄 Full End-to-End Flow

## ✅ Successful Session

1. Create session
2. Generate questions
3. Store answers
4. Evaluate via AI
5. Compute rubric averages
6. Compute risk score
7. Store final summary
8. Delete temporary data
9. Session finalized

---

## ❌ Terminated Session

1. Create session
2. Violations accumulate
3. Threshold exceeded
4. Session terminated
5. Malpractice snapshot stored
6. Delete questions & answers
7. Session finalized

No AI evaluation performed.

---

# 🚀 Scalability Considerations

* No per-frame logging
* No raw event storage
* Only aggregated counters
* JSONB used only for AI payloads
* Indexed session queries
* Atomic updates for concurrency safety

System designed to handle:

* 500+ concurrent sessions
* Burst violation updates
* Scalable Neon Postgres backend
